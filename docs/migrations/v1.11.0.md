# Migration to v1.11.0

This guide is **retroactive** — v1.11.0 shipped before the migration policy
existed (see [`README.md`](README.md)). The deprecation-warning
infrastructure that would have surfaced the change at compile time landed in
v1.12.0. Callers upgrading directly from v1.10.x to v1.11.x will see
`unknown intrinsic 'maddubs_i32'` rather than a warning.

## Summary

One breaking change: the polymorphic `maddubs_i32` was removed in favor of
explicit `maddubs_i16` + `madd_i16` chaining.

## Breaking changes

### Item: `maddubs_i32` removed (use `maddubs_i16` + `madd_i16`)

- **Why:** the old name hid a 2-instruction sequence behind one symbol,
  obscuring its cost and tying the API to the x86 pmaddubsw + pmaddwd
  fusion. The split form is portable to ARM (i8mm `usmmla_i32` for the
  equivalent fused operation) and makes the operation count visible.
- **Deprecation cycle:** none — shipped as a hard break in v1.11.0
  (commit `89130cb`). Future breaking changes will follow the one-minor-
  release deprecation policy.
- **Migration recipe (u8 × i8 → i32 dot, width 16/32/64):**

  ```ea
  // Before (v1.10.x):
  //   maddubs_i32(u8x16, i8x16) -> i32x4
  let acc = maddubs_i32(a, b)

  // After (v1.11.0+):
  //   maddubs_i16(u8x16, i8x16) -> i16x8
  //   madd_i16(i16x8, i16x8)    -> i32x4
  let lo  = maddubs_i16(a, b)        // u8×i8 pairwise multiply + add (sat)
  let acc = madd_i16(lo, splat<i16x8>(1)) // pairwise i16 -> i32 widening sum
  ```

  The `splat<i16x8>(1)` partner argument is what lets `madd_i16` collapse
  the i16 pairs into i32 lanes without changing the numeric result.

- **ARM equivalent:** when `--i8mm` is available, prefer `usmmla_i32` for
  the same operation; it does the full chain in a single instruction.

  ```ea
  // ARM Cortex-A76 with --i8mm:
  let acc = usmmla_i32(zero_i32x4, a, b)
  ```

- **Affects:** `Olorin` (gemma3 quant kernels), `eaclaw` (none — uses
  vdot_i32). Other downstream consumers should grep their `.ea` sources
  for `maddubs_i32`.

## Other v1.11.0 changes

Non-breaking additions are documented in
[`../../CHANGELOG.md`](../../CHANGELOG.md) under v1.11.0 — `--fp16`,
`--i8mm` (`smmla_i32` / `ummla_i32` / `usmmla_i32`), `exp_poly_f32`,
AVX-512 lane family, `f32x{4,8}_from_scalars`, full `cvt_f16_f32` widths.
None require source changes.

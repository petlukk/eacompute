# Migration to v1.12.0

See [`README.md`](README.md) for the deprecation-cycle policy.

## Summary

- **Monomorphic rename** for `sat_add` / `sat_sub` / `abs_diff` (polymorphic
  forms still work but emit a deprecation warning at every call site).
- Additive: `u64x{2,4,8}` vector types; `wmul_u64_lo` / `wmul_u64_hi`
  cross-platform widening multiply.
- Internal: deprecation-warning infrastructure, migrations directory,
  `cargo public-api` CI gate.

The breaking-removal cycle for the polymorphic names completes in
**v2.0.0** ‚Äî they remain callable throughout the v1.12.x / v1.13.x lifetime
with the warning. Update consumers when CI noise gets annoying enough; no
hard deadline before v2.0.

## Breaking changes (deprecation warnings, not removals)

### Item: `sat_add` / `sat_sub` / `abs_diff` ‚Üí typed spellings

- **Why:** the project standard is "concrete, not generic" ‚Äî every other
  intrinsic carries its element type in the name (`pack_sat_i16x8`,
  `wmul_u32`, `madd_i16`). The polymorphic three were remnants from before
  that rule was codified. Typed spellings make error messages, autocomplete,
  and search-and-replace land where the caller expects.
- **Deprecation cycle:** warning emitted from v1.12.0; removal scheduled
  for v2.0.0.
- **Migration recipe:**

  ```ea
  // Before:
  let x = sat_add(a, b)       // a, b: i8x16
  let y = abs_diff(p, q)      // p, q: u16x8

  // After:
  let x = sat_add_i8x16(a, b)
  let y = abs_diff_u16x8(p, q)
  ```

  Concrete spellings added in v1.12.0:

  | Polymorphic name | Typed spellings |
  | --- | --- |
  | `sat_add(a, b)` | `sat_add_i8x16`, `sat_add_u8x16`, `sat_add_i16x8`, `sat_add_u16x8` |
  | `sat_sub(a, b)` | `sat_sub_i8x16`, `sat_sub_u8x16`, `sat_sub_i16x8`, `sat_sub_u16x8` |
  | `abs_diff(a, b)` (ARM-only) | `abs_diff_i8x16`, `abs_diff_u8x16`, `abs_diff_i16x8`, `abs_diff_u16x8`, `abs_diff_i32x4`, `abs_diff_u32x4` |

  All typed spellings preserve the existing ARM-only / cross-platform
  semantics of the polymorphic parent.

- **Warning text emitted at the polymorphic call site:**

  ```
  warning: intrinsic 'sat_add' is deprecated since 1.12.0 ‚Äî use the typed
  spelling: sat_add_i8x16 / sat_add_u8x16 / sat_add_i16x8 / sat_add_u16x8
  (at line N, column N)
  ```

- **Affects:** `Olorin` (`sat_add`, `sat_sub` in several kernels), `Olorin`
  ARM-only paths (`abs_diff`), any future `.ea` consumer. Grep `.ea`
  sources in consumers and switch to the typed spellings at leisure
  before v2.0.0.

## Codegen bug-fix (incidental, no source change)

The polymorphic `sat_add` / `sat_sub` on x86 produced an undefined-symbol
link error when actually linked into an executable (`undefined reference
to llvm.x86.sse2.padds.b`). Existing tests only compiled to object file
so the bug went unnoticed in v1.11.0. Fix: switch the codegen to LLVM's
target-independent saturating intrinsics (`llvm.{s,u}{add,sub}.sat.vNiM`),
which the backend lowers to `padds`/`paddus`/`sqadd`/`uqadd` per target.
Removes the x86/ARM branching in `src/codegen/simd_saturating.rs`.

Same lesson as the `llvm.x86.sse2.pmulu.dq` issue we hit during the
`wmul_u64` work (PR #8): LLVM has been steadily moving from
target-specific intrinsic names to target-independent canonical forms.
Always link-test new x86 intrinsic codegen, not just compile-test.

## Additive changes (no migration needed)

Already landed in v1.12.0-dev:

- `u64x2` (128-bit, both platforms), `u64x4` (256-bit, AVX2), `u64x8`
  (512-bit, AVX-512) vector types over `u64`.
- `wmul_u64_lo(u32x4, u32x4) -> u64x2` + `wmul_u64_hi` ‚Äî unsigned widening
  multiply on the low/high halves. First cross-platform `wmul_*` family
  member. Unblocks Poly1305 5-limb radix-2¬≤‚Å∂ accumulator.

Planned for later in v1.12.x (not yet landed):

- `widen_u8_u16_hi(u8x16) -> u16x8` ‚Äî partner to existing `widen_u8_u16`.
- `cvt_f32_f16(f32x16) -> f16x16` ‚Äî width-16 form (AVX-512 only).
- `lo_i16x8(i16x16) -> i16x8` / `hi_i16x8` ‚Äî the lane family currently
  exposes lo/hi for i8/u8/i32/f32 but not i16.

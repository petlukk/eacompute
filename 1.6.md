# Eä v1.6 Language Specification

## Identity

Eä is a SIMD kernel compiler. Write compute kernels in explicit syntax, compile to native code, call from any language via C ABI. No runtime, no GC, no standard library. Just kernels.

**Eä is:** Branch-light, data-parallel, cache-aware numerics.
**Eä is not:** Bit-manipulation, parser-magic, or general-purpose programming.

The programmer sees the cost. If hardware can't do it, the compiler says so. If the code looks like SIMD, it produces SIMD — or fails at compile time. No silent scalar fallbacks, no hidden performance cliffs.

---

## Hard Rules

These are inviolable. Every change must respect them.

1. **No file exceeds 500 lines.** Split before you hit the limit.
2. **Every feature proven by end-to-end test.** If it's not tested, it doesn't exist.
3. **No fake functions. No stubs.** No `// TODO`, `// HACK`, `// for now`. If it does not compile and pass tests, it is not code.
4. **No premature features.** Don't build what isn't needed yet. Every feature must be motivated by a real kernel across real demos.
5. **Delete, don't comment.** Dead code gets removed, not commented out.
6. **C interop is the product.** Everything serves the goal of producing callable object files.
7. **Explicit over implicit.** No auto-masking, no silent scalar fallbacks, no hidden performance cliffs.
8. **Concrete, not generic.** No generics, no traits, no polymorphism. Write separate kernels for each type. Kernel code is monomorphic.
9. **One file, one compilation unit.** No modules, no imports. Compose at the C level.
10. **All memory is caller-provided.** No allocator, no GC. Pointers come from the host language.

---

## Design Philosophy

### How Eä Evolves

Eä follows a profile-driven evolution loop. Not feature-driven development. Not "Rust has X, we should too."

1. **Build a real kernel.** Write it in Eä, for a real workload, with real data.
2. **Measure on real hardware.** `perf stat`, cycles/row, IPC, cache behavior. Numbers, not intuition.
3. **Isolate the real bottleneck.** Branch mispredicts? Memory bandwidth? Instruction throughput? Name it.
4. **Accept limitations.** If the bottleneck is outside Eä's scope (OS, memory controller, algorithm), accept it. Not every problem needs a language feature.
5. **Change the language only if two independent demos benefit.** One kernel is an anecdote. Two kernels from different domains is a pattern. Hypothetical use cases don't count.

This is almost scientific method. Profile first, feature second, two proofs before merge.

### The Eä Test

Before adding any feature, it must pass all three:

1. **Does it serve data-parallel, cache-aware numerics?** If it's for string parsing, bit manipulation, or control-flow-heavy logic, it doesn't belong.
2. **Is it explicit?** The programmer must see the cost. No magic, no silent fallbacks.
3. **Is it motivated by real kernels?** At least two demos must benefit. Hypothetical use cases don't count.

### What Eä optimizes for

- **Throughput over latency.** Process vectors, not individual elements.
- **Predictable branches.** Branchless patterns via `select`, counted loops, uniform control flow.
- **Cache-aware data layout.** Flat arrays, struct-of-arrays, caller-controlled allocation.
- **Compile-time guarantees.** `static_assert`, const evaluation, type-checked SIMD widths.

### What Eä deliberately excludes

- **Scalar bitwise operators** (`&`, `|`, `>>`, `<<`, `^`) — these enable bit-manipulation patterns that pull the language away from data-parallel numerics. Use `% 2` and `/ 2` for the rare cases where bit extraction is needed, or use SIMD dot operators (`.&`, `.|`, `.^`) which operate on vector lanes.
- **Break/continue** — loops are counted or flag-controlled. This keeps control flow predictable.
- **Heap allocation** — all memory is caller-provided. The host language manages allocation.
- **String types** — Eä works on byte arrays via `*u8`. String semantics belong to the host.
- **Generics/polymorphism** — write concrete kernels for each type. The monomorphic guarantee means no hidden dispatch costs.

Element-wise vector operations use dot notation:

| Operator | Description |

|-----------|-----------|-------------|

| `stream_store(ptr, offset, val)` | `(*mut T, i32, VecT) → ()` | Non-temporal store (bypass cache) |

**Broadcast:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `splat(scalar)` | `(T) → VecT` | Broadcast scalar to all lanes |

**Reduction:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `reduce_add(vec)` | `(VecT) → T` | Horizontal sum of all lanes |
| `reduce_max(vec)` | `(VecT) → T` | Horizontal max of all lanes |
| `reduce_min(vec)` | `(VecT) → T` | Horizontal min of all lanes |

**Selection:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `select(mask, true, false)` | `(VecT, VecT, VecT) → VecT` | Branchless ternary per lane |

**Arithmetic:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `fma(a, b, c)` | `(VecT, VecT, VecT) → VecT` | Fused multiply-add: a*b+c |
| `sqrt(x)` | `(f32) → f32` | Scalar square root |
| `min(a, b)` | `(T, T) → T` | Scalar min (i32, f32, f64). CMOVcc/CSEL. |
| `max(a, b)` | `(T, T) → T` | Scalar max (i32, f32, f64). CMOVcc/CSEL. |

**Conversion:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `to_i32(x)` | `(u8\|i8\|f32) → i32` | Convert to i32 |

**Bit Extraction (x86 only):**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `movemask(vec)` | `(u8x16\|u8x32) → i32` | High bit of each byte lane → bitmask |

| Intrinsic | Signature | Description |
|-----------|-----------|-------------|

**Reorder:**
| Intrinsic | Signature | Description |
|-----------|-----------|-------------|
| `shuffle(vec, [indices])` | `(VecT, [i32]) → VecT` | Lane permutation |

### Control Flow

```
while condition { body }
if condition { body } else { body }
for i in 0..n { body }
for i in 0..n step 8 { body }
```

### Kernel Construct

```
kernel name(arr: *T, ...) range(0, n) {
    // body executes per-element, auto-vectorized by LLVM
}
```

Desugared to a `while` loop with tail handling before type checking. The `n: i32` parameter is auto-injected.

### Constants & Compile-Time

```
const NAME: type = value
static_assert(condition, "message")
```

### Structs

```
struct Name { field1: type, field2: type }
```

Scalar structs only. No SIMD aggregates.

### Functions

```
export func name(params) { body }          // exported (C ABI)
func name(params) -> ReturnType { body }   // private helper
#[cfg(x86_64)]
export func name(params) { body }          // conditional compilation
```

### Output Annotations (for binding generators)

```
out name: *mut T [cap: expr, count: path]
```

Marks parameters that bindings auto-allocate and return.

### Toolchain

```bash
ea kernel.ea                    # compile to .o
ea kernel.ea --lib              # compile to .so + .ea.json
ea kernel.ea --lib -o name.so   # named output
ea bind kernel.ea --python      # generate Python bindings
ea bind kernel.ea --rust        # generate Rust bindings
ea bind kernel.ea --cpp         # generate C++ bindings
ea inspect kernel.ea            # post-optimization instruction analysis
```

## v1.6 Features

Seven features, ordered by impact. All motivated by real kernel needs across 6+ demos (1BRC, eastat, eavec, easobel, astro_stack, video_anomaly).

### Feature 1: Scalar `min` / `max` Intrinsics

**Motivation:** Every demo with aggregation (1BRC, eastat, eavec) uses `if val < acc { acc = val }` — branch-heavy, mispredict-prone. The 1BRC microarchitectural analysis showed 25.6% of cycles wasted on branch mispredicts, with min/max updates as a primary source.

**Design:**
```
let a: i32 = min(x, y)
let b: f32 = max(x, y)
```

Scalar only. Accepts `i32` or `f32`. Returns same type.

**LLVM lowering:** `@llvm.smin.i32` / `@llvm.smax.i32` for integers (CMOVcc on x86, CSEL on ARM). `@llvm.minnum.f32` / `@llvm.maxnum.f32` for floats.

**Why this is Eä:** Branchless primitives for data-parallel reduction. Removes unpredictable branches from the hot path. The programmer explicitly says "min" — no magic, just the right instruction.

**Demos that benefit:** 1BRC (hash table min/max update), eastat (column min/max), eavec (top-K search).

**Effort:** ~60 lines (20 typeck, 20 codegen, 20 tests).

---

### Feature 2: Masked Load / Store

**Motivation:** Every SIMD kernel has a scalar tail loop: `while i < len { /* one element at a time */ }`. With masked load/store, the last partial vector can be processed with a single masked SIMD operation, eliminating the tail loop entirely.

**Design:**
```
let v: f32x8 = load_masked(ptr, offset, mask, passthru)
store_masked(ptr, offset, mask, value)
```

`mask` is a vector of the same width with all-ones (active) or all-zeros (inactive) per lane. `passthru` provides default values for inactive lanes on load.

**LLVM lowering:** `llvm.masked.load` / `llvm.masked.store` — these are target-independent LLVM intrinsics that lower to optimal code on both x86 (AVX2 `vmaskmovps`) and ARM (SVE masked loads, NEON with blend).

**Why this is Eä:** Eliminates the most common source of code duplication in kernels. The programmer explicitly controls which lanes are active. No auto-masking magic — you build the mask, you pass it in.

**Demos that benefit:** All 6 demos. Every kernel with a `while i + 8 <= len` / `while i < len` pattern.

**Effort:** ~120 lines (30 typeck, 50 codegen, 40 tests).

---

### Feature 3: Conditional Compilation

**Motivation:** ARM and x86 kernels often need different implementations. Currently this requires separate files (`scan.ea` vs `scan_arm.ea`, `csv_scan.ea` vs `csv_scan_arm.ea`, `csv_stats.ea` vs `csv_stats_arm.ea`). This doubles maintenance, duplicates shared logic, and makes `build_kernels.sh` scripts more complex.

**Design:**
```
#[cfg(x86_64)]
export func scan_fast(text: *u8, len: i32, out: *mut i32) {
    // AVX2 movemask path
}

#[cfg(aarch64)]
export func scan_fast(text: *u8, len: i32, out: *mut i32) {
    // NEON scalar-extract path
}
```

Attribute on functions. Evaluated at compile time based on target triple. Same function name, different implementations — the compiler selects the right one.

**Supported targets:** `x86_64`, `aarch64`. Future: `avx512` (when `--avx512` flag is set).

**Why this is Eä:** One file, one compilation unit — but with explicit platform selection. The programmer sees both implementations. No hidden fallbacks. The `#[cfg]` attribute is explicit and visible at the function level.

**Demos that benefit:** 1BRC (scan.ea + scan_arm.ea → one file), eastat (csv_scan.ea + csv_scan_arm.ea → one file, csv_stats.ea + csv_stats_arm.ea → one file).

**Effort:** ~150 lines (40 parser, 30 typeck, 40 codegen, 40 tests).

---

### Feature 4: `f64x4` / `f64x2` Vector Types

**Motivation:** Statistical kernels (eastat, eavec) accumulate millions of values. `f32` has ~7 decimal digits of precision — for large sums, accumulated error becomes significant. Kahan summation in `f32` works but adds complexity. `f64` accumulators with `f32` input/output is the standard approach.

**Design:**
```
let acc: f64x4 = splat(0.0) : f64x4     // x86 AVX2: 4 × 64-bit
let acc: f64x2 = splat(0.0) : f64x2     // ARM NEON: 2 × 64-bit
```

All existing intrinsics (`load`, `store`, `splat`, `reduce_add`, `fma`, `select`, dot operators) work on `f64x4` and `f64x2`.

Add conversion intrinsic:
```
let wide: f64x4 = cvt_f32_f64(narrow_f32x4)   // f32x4 → f64x4, lossless
let narrow: f32x4 = cvt_f64_f32(wide)          // f64x4 → f32x4, rounding
```

**LLVM lowering:** `<4 x double>` on x86 AVX2, `<2 x double>` on NEON. `fpext`/`fptrunc` for conversions.

**Why this is Eä:** More precision for data-parallel numerics. The programmer explicitly chooses `f64x4` when accuracy matters — no automatic promotion. The cost is visible: half the throughput of `f32x8`.

**Demos that benefit:** eastat (column statistics with f64 accumulators), eavec (cosine similarity with f64 intermediate sums).

**Effort:** ~200 lines (codegen infrastructure already handles vector types generically; main work is type checking + conversion intrinsics + tests).

---

### Feature 5: `i32x8` / `i32x4` Vector Types

**Motivation:** Integer SIMD operations appear in hash computation (1BRC), index arithmetic (astro_stack), and threshold counting (video_anomaly). Currently these use scalar `i32` loops. `i32x8` would allow 8-wide integer vectorization for hash buckets, index computation, and integer accumulation.

**Design:**
```
let v: i32x8 = load(indices, i) : i32x8
let sum: i32 = reduce_add(v)
```

Supports: `load`, `store`, `splat`, `reduce_add`, `select`, `.+`, `.-`, `.*`, `.==`, `.<`, `.>`.

Does NOT support: `./` (integer division is not SIMD-friendly), `.&`/`.|`/`.^` (Eä avoids bitwise on integers — use `select` for conditional patterns).

**LLVM lowering:** `<8 x i32>` on x86 AVX2, `<4 x i32>` on NEON (as `i32x4`, already partially supported).

**Why this is Eä:** Data-parallel integer arithmetic. Hash tables, index arrays, and counters are core to Eä workloads. The restriction against integer bitwise dot operators keeps Eä true to its identity — no bit-manipulation patterns creeping in through vector types.

**Demos that benefit:** 1BRC (vectorized hash computation), astro_stack (index arithmetic), video_anomaly (threshold counting).

**Effort:** ~180 lines (type system + codegen + tests). Much of the infrastructure exists from `f32x8`.

---

### Feature 6: `for` Loop Syntax

**Motivation:** Every kernel has `let mut i: i32 = 0; while i < n { ... i = i + 1 }`. This is 3 lines of boilerplate for the most common pattern. A `for` loop is clearer, less error-prone, and communicates intent better.

**Design:**
```
for i in 0..n {
    // i is i32, immutable within body
}

for i in 0..n step 8 {
    // stride of 8
}
```

Desugared to `while` loop in `desugar.rs`, exactly like `kernel` → `func` transformation. `i` is an implicitly-declared `i32`. The range is half-open `[start, end)`.

**Not included:** `for` over arrays, iterators, or collections. Eä is not a general-purpose language. This is a counted loop, nothing more.

**Why this is Eä:** Reduces boilerplate without adding magic. The semantics are identical to the `while` loop it replaces. The desugarer does the transformation — type checker and codegen never see it.

**Demos that benefit:** All 6 demos. Every kernel with a counted loop.

**Effort:** ~100 lines (30 parser, 30 desugar, 40 tests). Follows the `kernel` desugaring pattern exactly.

---

### Feature 7: Pointer-to-Pointer Parameters

**Motivation:** Batch operations (astro_stack `accumulate_batch`, potential batch convolution) need to pass arrays of image pointers. Currently this requires flattening into a single buffer or using index arithmetic. `**T` allows clean signatures for multi-buffer operations.

**Design:**
```
export func accumulate_batch(
    frames: **f32,        // array of pointers to f32 arrays
    n_frames: i32,
    width: i32,
    out: *mut f32
) {
    // frames[k] gives *f32 pointing to frame k's data
}
```

Only one level of indirection: `**T` (pointer to pointer). No `***T`.

**LLVM lowering:** `ptr` (opaque pointer). Array indexing on `**T` returns `*T`. No new LLVM concepts needed.

**Why this is Eä:** Cleaner C interop for real batch workloads. The alternative (flattening all frames into one buffer with stride arithmetic) is error-prone and hides the data structure. `**T` is what C code already uses for these patterns.

**Demos that benefit:** astro_stack (batch frame accumulation), potential batch convolution patterns.

**Effort:** ~100 lines (20 parser, 30 typeck, 30 codegen, 20 tests).

## Version History

### v1.6 (current)

Profile-driven consolidation release. Every feature motivated by `perf stat` analysis across 6 demos.

- **Scalar `min`/`max` intrinsics:** `llvm.smin`/`llvm.smax` for i32, `llvm.minnum`/`llvm.maxnum` for f32/f64. Branchless CMOVcc/CSEL. Attacks the 25.6% branch mispredict tax from 1BRC analysis.
- **`f64x4` / `f64x2` vector types:** Double-precision SIMD for statistical accumulators. f64x4 on x86 AVX2, f64x2 on ARM NEON.
- **`i32x8` verification:** Token/parser already existed. Full operation coverage confirmed (load, store, splat, reduce, select, arithmetic).
- **`for` loop syntax:** `for i in 0..n step 8 { }`. Desugared to while loop, same as kernel→func pattern.
- **`#[cfg(x86_64|aarch64)]` conditional compilation:** Merge platform-specific kernels into one file. Evaluated at desugar time via `filter_cfg()`.
- **Pointer-to-pointer `**T`:** Recursive parse_type() handled it out of the box. Clean batch signatures for multi-buffer operations.
- **420 tests**, all files ≤500 lines, clippy/fmt clean.

### v1.5

- **Multi-kernel files:** Multiple exports, shared structs, shared consts in one file. Always worked — documented and claimed.
- **`static_assert`:** Compile-time assertions with const evaluation. Top-level only, evaluated at type-check time.
- **`ea inspect`:** Post-optimization instruction analysis. Shows vector/scalar instruction counts, vector widths, loop counts, register usage.
- **`movemask` intrinsic:** `u8x16`/`u8x32` → `i32` bitmask. x86 only. Enables SIMD structural scanning.

### v1.4

- Core language: functions, kernels, structs, consts.
- SIMD types: `f32x8`, `f32x4`, `u8x16`, `u8x32`, `i8x16`, `i16x8`.
- Intrinsics: `load`, `store`, `stream_store`, `prefetch`, `splat`, `reduce_add`, `reduce_max`, `reduce_min`, `select`, `fma`, `sqrt`, `to_i32`, `to_f32`, `maddubs_i16`, `maddubs_i32`, `shuffle`.
- Binding generators: Python, Rust, C++, PyTorch, CMake.
- Output annotations with auto-allocation in bindings.
- Targets: x86-64 (AVX2 default, AVX-512 via `--avx512`), AArch64/NEON.

## Microarchitectural Baseline (AMD EPYC 9354P, Zen 4)

From 1BRC `parse_aggregate` kernel (10M rows, 131 MB):

```
125 cycles/row, IPC 1.79, 303 MB/s
Branch mispredicts: 25.6% of cycles (primary bottleneck)
Hash table (84KB): L2-resident, 1.5% bandwidth used
Data stream: L3 at 303 MB/s, well below peak
```

This is the target the v1.6 features aim to improve. Scalar `min`/`max` directly attacks the 25.6% branch mispredict tax. Masked load/store eliminates tail-loop branches. `i32x8` could vectorize the hash computation chain.

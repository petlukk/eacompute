# Sobel Edge Detection — Ea Demo

Four-tool comparison: Ea vs OpenCV vs scikit-image vs NumPy.

All implementations compute Sobel gradient magnitude on grayscale f32 images.
Ea and NumPy use L1 norm (`|Gx| + |Gy|`). scikit-image uses L2 norm
(`sqrt(Gx² + Gy²)`). OpenCV uses L1 norm but with different border handling.

Zero manual ctypes — Python bindings auto-generated by `ea bind --python`.

## Results (single-threaded)

AMD Ryzen 7 1700 (8C/16T, 3.0 GHz), 8 MB L3, WSL2. 50 runs, 10 warmup, median.
All implementations single-threaded. OpenCV pinned to 1 thread (`cv2.setNumThreads(1)`).

```
  Size       Pixels        Ea    OpenCV     NumPy  vs cv2   vs np   Mpx/s
────── ──────────── ───────── ───────── ───────── ─────── ─────── ───────
  720p      921,600    1.26ms    6.58ms   15.13ms    5.2x   12.0x    732
 1080p    2,073,600    3.68ms   15.03ms   36.50ms    4.1x    9.9x    563
    4K    8,294,400   12.55ms   59.72ms  172.84ms    4.8x   13.8x    661
```

The single-threaded kernel is the primary result. The Ea kernel at 1.26ms for
720p is 5× faster than OpenCV's single-threaded path and 12× faster than NumPy.
This is a **kernel efficiency** comparison — same algorithm, same core count,
measuring what happens inside one core with explicit SIMD vs generic code.

## Multi-threaded results

The demo supports row-striped multi-threading (`--threads N`). Each thread
runs the same SIMD kernel on a horizontal strip; ctypes releases the GIL so
threads run truly in parallel.

Same machine, 16 threads:

```
  Size       Pixels        Ea     Ea MT    OpenCV     NumPy   MT/ST  vs cv2   vs np   Mpx/s
────── ──────────── ───────── ───────── ───────── ───────── ─────── ─────── ─────── ───────
  720p      921,600    1.26ms    2.21ms    6.58ms   15.13ms    0.6x    3.0x    6.8x    417
 1080p    2,073,600    3.68ms    3.07ms   15.03ms   36.50ms    1.2x    4.9x   11.9x    675
    4K    8,294,400   12.55ms    8.48ms   59.72ms  172.84ms    1.5x    7.0x   20.4x    978
```

At 720p (1.26ms ST), thread dispatch overhead exceeds kernel time — the 0.6×
result is the correct one. The crossover is at 1080p (1.2×), and at 4K the MT
path reaches 1.5× over ST. On native hardware with better thread scheduling
than WSL2, MT scaling would improve further.

The MT column is interesting but secondary. Ea's value is that the
single-threaded kernel is already faster than OpenCV's single-threaded path by
a significant margin. Multi-threading is additional throughput on top of an
already-efficient kernel.

## Why single-threaded OpenCV

The comparison pins OpenCV to 1 thread because Ea runs one kernel on one core.
This is a **kernel efficiency** comparison, not a parallelism comparison.

On a multi-core machine, OpenCV with all threads enabled would close the gap
significantly. On an 8-core Ryzen, multi-threaded OpenCV could run 4-6x faster
than single-threaded, potentially matching the Ea kernel's single-core
throughput. The speedup measures what happens inside a single core: the Ea
kernel uses explicit f32x4 SIMD with prefetch, while OpenCV's generic code
path carries more overhead per pixel.

## L1 vs L2 norm

The 0.24 pattern difference between Ea and scikit-image is not an error — it's
a genuine algorithmic difference:

- **Ea, OpenCV, NumPy**: L1 norm — `|Gx| + |Gy|`
- **scikit-image**: L2 norm — `sqrt(Gx² + Gy²)`

Same Sobel gradients, different magnitude combination. For edge detection the
results are visually nearly identical (edges appear in the same places), but
numerically they diverge. L1 is cheaper to compute (no square root) and
produces slightly sharper edges. L2 is rotationally invariant.

Ea vs NumPy: exact match (max diff = 0.000000). Both use identical L1 Sobel.
Ea vs OpenCV: exact match on interior pixels. Border pixels differ due to
OpenCV's reflect border vs Ea's zero padding.

## The kernel

75 lines of Ea (`sobel.ea`). Stencil pattern: f32x4 SIMD inner loop with
software prefetch, scalar tail loop for remainder pixels.

```
export func sobel(input: *restrict f32, out: *mut f32, width: i32, height: i32)
```

- **SIMD loop**: loads 3 rows × 3 columns as f32x4 vectors, computes Gx and
  Gy via dot-operator arithmetic, branchless abs via `select`, stores 4 pixels
  per iteration
- **Scalar tail**: handles remaining 1-3 columns with scalar arithmetic
- **Prefetch**: `prefetch(input, row + x + 16)` for all three rows, staying
  ahead of the access pattern
- **Zero border**: output border pixels are left as zero (same as NumPy)

Verified against reference implementations on the Kodak benchmark image

## How to run

```bash
# Build compiler (once)
cargo build --features=llvm --release

# Build kernel + generate bindings
bash demo/sobel/build.sh

# Run demo
python3 demo/sobel/run.py                # full benchmark (720p, 1080p, 4K)
python3 demo/sobel/run.py --quick        # 720p only
python3 demo/sobel/run.py --threads 4    # multi-threaded with 4 threads
python3 demo/sobel/run.py photo.jpg      # benchmark on a specific image
```

Requires: Python 3, NumPy, Pillow.
Optional: OpenCV (`pip install opencv-python`), scikit-image (`pip install scikit-image`).
Downloads Kodak benchmark image on first run.

## Build artifacts

`build.sh` runs two commands:

```bash
ea sobel.ea --lib -o libsobel.so     # compile kernel to shared library
ea bind sobel.ea --python            # generate sobel.py (NumPy + ctypes)
```

The generated `sobel.py` handles ctypes setup, dtype checking, and pointer
conversion. `run.py` imports it as a normal Python module — zero manual FFI.
